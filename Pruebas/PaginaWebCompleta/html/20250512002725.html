T2WML: Table To Wikidata Mapping Langauge
Pedro Szekely, Daniel Garijo, Divij Bhatia, Jiasheng Wu, Yixiang Yao and Jay Pujara
szekely@usc.edu,dgarijo@isi.edu,divijbha@usc.edu,jiashenw@usc.edu,yixiangy@isi.edu,jpujara@usc.edu
USC Information Sciences Institute

ABSTRACT
The web contains millions of useful spreadsheets and CSV les, but
these les are dicult to use in applications because they use a
wide variety of data layouts and terminology. We present Table To
Wikidata Mapping Language (T2WML), a language that makes it
easy to map and link arbitrary spreadsheets and CSV les to the
Wikidata data model. The output of T2WML consists of Wikidata
statements that can be loaded in the public Wikidata knowledge
base or in a Wikidata clone repository, creating an augmented
Wikidata knowledge graph that application developers can query
using SPARQL.

including data from many government and international organi-
zations. Most institutions oer their data in web sites where users
can download the data in Excel and CSV formats. The downloaded
data is seldom directly usable because, unlike databases (which use
one column per variable), spreadsheets often arrange the data in
dierent layouts.

Existing languages and systems for mapping structured data to
RDF, including R2RML,
3
RML [
1

3
] and CSV2RDF [
2
] are
row based, designed for database layouts (Fig. 1c). The interpreters
for these languages process one row at a time, using the column
headers to dene the properties used for generating RDF (RML sup-
ports other formats such as JSON and XML). These languages also
provide preprocessing operations to dene new columns, to join
multiple tables and to fold/unfold columns and rows to rearrange
data for row-based processing. The preprocessing transformations
to convert the data in tables a), b) and d) to the database layout
are beyond the processing capabilities of existing RDF mapping
systems, and external scripting would be required to map these
les to an ontology to generate RDF.
T2WML is a mapping language designed to meet three objectives:
(1)
Map data in arbitrary data layouts used in Excel and CSV les
without the need of complex preprocessing steps to transform tables
into a canonical "Database" representation.
(2)
Enable users who
are not familiar with RDF to map spreadsheets and CSV les to
knowledge graphs so that they can augment knowledge graphs with
data useful in downstream applications.
(3)
Integrate mapping and
entity linking so that the resulting output is linked to a reference
knowledge graph, avoiding the need for a separate linking process
that is often neglected in the current transformation workows.
T2WML is designed for the Wikidata data model [4]. The main
building block in this model is a statement, which consists of a sub-
ject, a predicate, an object, qualiers and references. The subject,
predicate an object part mirror their RDF counter parts. The qual-
iers are predicate/object pairs that provide context information
about a subject/predicate/object triple. For example, qualiers can
be used to qualify an employment relation between a person and
an organization to record the period of time when the person was

is used to record sources from where the knowledge was derived

then introduce the language specication and system, and nally
discuss our experience using it to extend Wikidata with data in
spreadsheets downloaded from multiple sites.
2 T2WML DESIGN
T2WML is a template based language that models spreadsheets
and CSV les using two concepts.
Data blocks
: possibly non-
contiguous blocks of cells containing values of interest (green cells
in Fig. 1). These blocks can occur anywhere in a table.
Context
:
cells that describe the values in data blocks (blue cells in Fig 1).
The meaning of a value in a data block is dened using the val-
ues in selected context cells. For example, the value highlighted in
Fig. 1a represents the number of homicides of males in Burundi in
the year 2000 as reported by SDG. The context cells are spatially
related to the target value cell, but in each table the spatial arrange-
ment is dierent. For example, in Fig. 1a the country, source and
population are in the same row, and the year is in the top row above
the homicides number.
T2WML denes an expressive, yet simple language to identify
the context cells for a target value cell. This cell-centric model gives
T2WML the expressive power lacking in row-centric models in the
languages derived from R2RML. In R2RML, a mapping statement
can refer to cells in the same row, while T2WML can refer to cells
in distant rows. For example, in Fig. 1b the population indicators
(male, female) are in the rst row, and the years are in the second
row. In Fig. 1d the situation is more complex as the year cells are
not identied by absolute row coordinates, but by their geometric
arrangement relative to the target cell (i.e., in the nearest row above
the target cell containing a value in the rst column and empty
values in the rest of the columns).
A second dierence with existing mapping tools is that T2WML
adopts a “link rst” strategy where cells that identify entities in the
target knowledge graph (Wikidata in our implementation) must
be linked to the corresponding entities before the input data is pro-
cessed to produce statements in the knowledge graph. The link-rst
strategy oers multiple benets over the traditional link-later strat-
egy used in existing tools. It ensures that linking is done, resulting
in higher quality data, and at the same time avoids the need for
users to design a URI scheme. Cells can also be linked to prop-
erty nodes in Wikidata, enabling T2WML to also contextualize the
property used to describe a value.
T2WML integrates a cell linking service (a cell wikier). Users
can submit cell ranges for wikication and curate the results within
the T2MWL user interface (Wikier panel in Fig. 2). In addition,
T2WML is interoperable with the tools submitted to the 2019 ISWC
Cell Annotation Challenge
4
).
A third dierence with existing mapping tools is that T2WML
uses YAML
5
instead of RDF to represent mappings. Our choice
of YAML is deliberate as YAML has become a popular language
among developers for representing structured data. It is easy to edit
and easy to generate from software, enabling T2WML to be easily
incorporated in larger software pipelines. The T2WML approach,
is RDF friendly, as the processor can generate Wikidata statements
in RDF, but is also accessible to developers and users unfamiliar
with RDF (statements may also be generated in JSON-LD).
3 T2WML LANGUAGE
In this section we describe the most important features of the
T2WML language. The formal specication of the full language is
available in https://github.com/usc-isi-i2/t2wml.
Denitions. The key concept in T2WML is a cell, identied using
two variables,
$col
and
$row
.
$col
ranges from column
A
until the
last non-empty column, and
$row
is dened similarly. All constructs
in the language are dened as functions of $col and $row.
Denition: (row and column expressions) A row expression is a
function f ($col, $row) whose value identies a row in a table.

Simple expressions include constants such as
3
, which identi-
es the third row regardless of the values of
$col
and
$row
, and
arithmetic expressions such as
$row-2
, which identies the row
two rows above the current one (e.g., if
$row = 3
then
$row-2
designates row 1).
Complex expressions can be dened using the
->
operator which
takes two arguments, a Boolean expression and A simple expression.
For example,
value(A/$row) = "hello" -> $row+1
produces
the row below if the cell in the current row in column
A
contains
the value
hello
. If the Boolean expression is false the value is
undened. The second argument is optional allowing for a terse
syntax. For example,
value(A/$row) = "hello"
is a shortcut for
value(A/$row) = "hello" -> $row
if the expression is used in
a context where the value is a row.
Denition: (cell expression) A function f(column expression, row
expression) whose value identies a cell in a table. The
/
(slash)
operator denes cell expressions, combining a column expression
and a row expression. For example,
C/$row-2
identies the cell in
column
C
two rows above the current row. The
->
is overloaded
and can be used to construct cell expressions, in addition to column
or row expressions.
Denition: (value(cell expression)) Retrieves the value of a cell
in a table. Fig. 2 shows examples of value expressions in lines 5, 7,
9, 13 and 16. For example, value(
$col
/
$row
) (line 9) identies the
value of the current cell.
Denition: (item(cell expression)) Retrieves the identier of the
item linked to the cell identied using the cell expression. The
T2WML system provides multiple ways for users to dene the
behavior of the
item()
function. Users can invoke any wikier
service that returns results as annotations on cells, and they can
also curate or dene the mapping of cells to identiers in the user
interface. Fig. 2 shows examples of item expressions in lines 11, 21
and 23.
Denition: (iteration expression) Denes a sequence of cell ref-
erences to iterate over until a Boolean expression returns true.
Iteration expressions are dened using the
-n
or
+n
operators. The
iteration starts with
n=0
and increments by one until the containing
Boolean expression returns true or the cell reference containing
it abandons the connes of a table. For example, in the table in
Fig. 1d, the expression
value(A/$row-n)
can be used to retrieve
the country that denes the context of any of the homicide counts.
3.1 T2WML Statement Mappings
We describe the language for mapping tables to statements using
Fig. 2, which shows the user interface for the T2WML system. The
panel on the left shows the table in Fig. 1a, and the YAML Editor
panel shows a T2WML specication to map this data to Wikidata.
T2WML specications consist of a data
region
to identify the
cells containing the values of statements (e.g., homicide counts,
highlighted in green in the Table Viewer in Fig. 2), and a
template
section to dene the statements for each cell in the data region.
T2WML denes data regions using column and row expressions.
Data regions conne an iterator to visit all cells containing data
values. In our example, the data region is dened by columns
C
and right edge of the table (the default) and rows
4
and
14
. Using
constant row and cell expressions yields correct results for this table,
but we use a predicate to dene the bottom edge, as the number of
countries can change, and footnotes may be present at the bottom
of the table. The predicate
bottom: value(B:/$row) = ""
states
that the bottom edge is the rst row where columns after
B
are
empty.
6
The example also illustrates the ability to skip rows and
cells using Boolean expressions. The cells in the data region are
highlighted in green.
The
template
section denes the mapping of cell to elements of a
statement. The T2WML tool instantiates the template once for every
cell dened in the
region
section, binding the variables
$col
and
$row
to the coordinates of the cell being processed. To facilitate
understanding of the template instantiation procedure, users can
click on a data cell in the table viewer to see how it is mapped.
The interface shows the values of
$col
and
$row
, highlights the
cell containing the item (subject) of the statement (blue), the cells
containing the qualiers (pink), and shows the resulting statement
in an output panel (bottom right of Fig. 2).
Users dene the relationships between a value cell and other
parts of a statement in the YAML editor using expressions dened
above. Fig. 2 illustrates the denition of the subject, predicate, value
and qualiers of a statement, which we summarize below:
Subject: line 11 denes the subject of the statement for a value
cell as the item in the same row in column A. For the value in cell
F6, the item is Burundi, also shown in the output panel.
Predicate: the predicate of the statement is specied as a constant
(
P100024
, dened in our clone of Wikidata). It is also possible to
dene the properties using the
item
function, a convenient feature
to map spreadsheets where dierent columns contain information
about dierent properties.
Value: the value of a statement is usually dened using the ex-
pression
value($col/$row)
, or function that transforms the value.
T2WML oers a library of string, data and numeric transformation
functions similar to those provided in spreadsheet software.
Qualiers: the qualiers of a statement are dened in the
qualifiers
section of the YAML le. Each qualier consists of
a predicate and a value. Lines 15 through 19 illustrate the denition
of a time qualier including specication of the value, calendar,
precision and time zone.
References: references are dened in the
references
section of
the YAML le (not shown in the gure), and are dened similarly
to qualiers.

4 DISCUSSION
T2WML is a new language and system under active development.
While our experience with T2WML is limited, the results so far are
encouraging.
We evaluated the expressivity of T2WML by creating 19 variants
7
of the homicides table downloaded from dataunodc.un.org. We
created the variants from the Database layout (Fig. 1c), moving the
qualiers into header rows to emulate common layouts, including
multiple header rows (Fig. 1b). We created stacked table variants
(Fig. 1d) as tables of this type are also common in web sites. T2WML
can map all variants except one where we combined the female and
male homicide values in a single cell, separated by comma.
We used T2WML to create Wikidata statements for 9 World
Bank indicators (all countries, all available years), and for the Frag-
ile States Index indicators (https://fragilestatesindex.org). Political
scientists are using these statements to build models querying our
Wikidata clone.
We also used T2WML to map county-level crime data from the
FBI.
8
We had originally written a 400-line Python script to map this
data, and were able to replicate the results using a T2WML le of
the same complexity as the one shown in Fig. 2.
Finally, we evaluated the usability of T2WML with a second-year
undergraduate student who is creating Wikidata statements for
public crime records in Los Angeles. The dataset contains over 2
million records and is updated weekly. Each crime record is highly
detailed, including information such as the address, type of crime,
time of day and location, source, etc. Despite not being an expert in
mapping languages or RDF, the student was able to use T2WML to
link crime records to Wikidata locations using Wikidata properties.
Our current and future work will focus on four directions. First,
T2WML can create statements, but currently cannot create new
Wikidata items or properties, or dene new labels, aliases and de-
scriptions. We will extend T2WML to support these tasks. Second,
we want to integrate T2WML more tightly with Wikidata to sup-
port semantic operators to enable users to refer to elements of a
table semantically (e.g., the column containing countries). Third,
we are investigating machine learning approaches for property rec-
ommendation as we found that identifying the correct properties
to use is the most dicult part of the mapping process. Finally, we
are investigating ideas for publishing T2WML les in Wikidata
to crowdsource the creation of mapping les for the signicant
number of spreadsheet and CSV les that exist on the web.
